# Digest de Pesquisa — Planejamento refactor-arch (2026-06-29)

Este digest consolida os achados de cinco frentes de pesquisa para fundamentar uma skill de auditoria/refatoração arquitetural (MBA). Cada seção traz uma introdução curta, os achados-chave com a fonte inline e, quando aplicável, uma tabela de mapeamento depreciado→moderno. A seção final transforma tudo em diretrizes acionáveis para o catálogo de anti-padrões, o playbook de refatoração, as restrições de autoria da skill e a arquitetura-alvo MVC.

## Python 3.12+ e Flask 3.x — práticas atuais e substituições de APIs depreciadas

Foco em modernizar um app Flask legado para MVC: data/hora timezone-aware, remoção de APIs apagadas no Werkzeug/Flask, padrão application-factory + Blueprints, SQL parametrizado, hashing de senhas e gestão de segredos/debug.

- `datetime.utcnow()` e `datetime.utcfromtimestamp()` estão depreciados desde o Python 3.12; a doc oficial afirma **apenas depreciação**, sem versão de remoção confirmada ("Deprecated since version 3.12: Use datetime.now() with UTC instead."). A causa-raiz é que `utcnow()` retorna um datetime *naive* tratado por muitos métodos como hora local; os substitutos retornam objetos *aware* (`datetime.now(timezone.utc)` e `datetime.fromtimestamp(ts, tz=timezone.utc)`). ([docs.python.org/datetime](https://docs.python.org/3/library/datetime.html))
- Armadilha de migração em apps Flask/SQLAlchemy legados: bancos e o `DateTime` padrão do SQLAlchemy (sem `timezone=True`) armazenam UTC *naive* — a tzinfo é descartada na escrita e precisa ser reanexada na leitura. A refatoração correta **ainda** é sair de `utcnow()` para `datetime.now(timezone.utc)`; o detalhe de storage é nota de rodapé (use `DateTime(timezone=True)` ou normalize no load), não motivo para manter `utcnow()`. ([blog.miguelgrinberg.com](https://blog.miguelgrinberg.com/post/it-s-time-for-a-change-datetime-utcnow-is-now-deprecated))
- O Werkzeug 3.x removeu um grande lote de APIs legadas (`werkzeug.urls.url_decode/url_encode`, `format_string`, `escape/unescape`, `detect_utf_encoding`, o módulo de filesystem); o Flask 3.1 exige Werkzeug ≥ 3.1, ItsDangerous ≥ 2.2 e Blinker ≥ 1.9. Código legado que importa `url_decode/url_encode` (ou depende de flask-login/flask-wtf antigos) lança `ImportError` no Werkzeug moderno. ([werkzeug.palletsprojects.com/changes](https://werkzeug.palletsprojects.com/en/stable/changes/))
- O próprio Flask removeu APIs legadas comuns em 2.3/3.0: `@app.before_first_request` / `@bp.before_app_first_request`, importar `escape`/`Markup` de `flask`, `app.json_encoder`/`json_decoder` e `flask.json.JSONEncoder/JSONDecoder`, e `flask.__version__`. ([flask.palletsprojects.com/changes](https://flask.palletsprojects.com/en/stable/changes/))
- A estrutura idiomática do Flask 3.x para separação MVC é o padrão application-factory + Blueprints, com extensões ligadas via `init_app()`: um `create_app(config)` constrói o app, carrega config, chama `ext.init_app(app)` em singletons criados no escopo do módulo (ex.: `db = SQLAlchemy()`) e registra Blueprints. Mapeamento MVC: view functions de Blueprint = controllers (só HTTP), camada `services/` = regra de negócio, `models/` = dados (SQLAlchemy). ([flask.palletsprojects.com/appfactories](https://flask.palletsprojects.com/en/stable/patterns/appfactories/))
- Blueprints são o mecanismo oficial para modularizar rotas/views — "make application components and support common patterns" e "greatly simplify how large applications work" — registrados no factory com `app.register_blueprint(bp)`. ([flask.palletsprojects.com/blueprints](https://flask.palletsprojects.com/en/stable/blueprints/))
- Nunca montar SQL com operações de string; usar placeholders do DB-API. A doc do `sqlite3` avisa textualmente: "beware of using Python's string operations to assemble queries, as they are vulnerable to SQL injection attacks" e "Always use placeholders instead of string formatting". Estilos: qmark (`... WHERE id = ?`, `(value,)`) e named (`... :id`, `{'id': value}`). Vale para `%`, `.format()` e f-strings. ([docs.python.org/sqlite3](https://docs.python.org/3/library/sqlite3.html))
- Para hashing de senhas, a primeira escolha da OWASP (2024+) é Argon2id (mínimo m=19 MiB, t=2, p=1); scrypt (N=2^17, r=8, p=1), bcrypt (work factor ≥ 10, cap de 72 bytes) e PBKDF2 (≥ 600.000 iterações, só FIPS-140) são alternativas condicionais. ([cheatsheetseries.owasp.org/Password_Storage](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html))
- `werkzeug.security.generate_password_hash` usa scrypt por padrão (desde 2.3; `method="scrypt"`, n=2^15, r=8, p=1; PBKDF2 default subiu para 1.000.000 iterações). O Werkzeug **não** embute Argon2 — para a primeira escolha da OWASP use `argon2-cffi` (`argon2.PasswordHasher`). Em qualquer caso, armazene só o hash codificado e verifique com o helper de tempo constante. ([github.com/pallets/werkzeug/security.py](https://github.com/pallets/werkzeug/blob/main/src/werkzeug/security.py))
- Debug mode deve ser `False` em produção (o debugger interativo "allows executing arbitrary Python code" no servidor) e dirigido por `FLASK_DEBUG`, nunca `app.run(debug=True)` hardcoded. `SECRET_KEY` deve vir de variável de ambiente (`secrets.token_hex()`), via `from_object(...)` e/ou `from_prefixed_env()`, com `python-dotenv` só em dev; adicione assertions de startup. ([flask.palletsprojects.com/config](https://flask.palletsprojects.com/en/stable/config/))

## Node.js e Express — práticas atuais e substituições de APIs depreciadas

Foco em sair do estilo callback (sqlite3 callbacks, crypto caseira, god-class) para MVC: async/await, Express 5, driver de SQLite com prepared statements, Argon2id, redação de logs e camada de segurança (helmet, rate limit).

- Migrar a pirâmide de callbacks para async/await; no Express 5 não é mais preciso `.catch(next)` porque promises rejeitadas são auto-encaminhadas para o error middleware. Use `util.promisify()` (Node 8+) para envolver APIs error-first, ou prefira a API de promises nativa da lib. O padrão `getUserById(id).then(...).catch(next)` colapsa para `async (req,res) => { const u = await getUserById(id); res.send(u); }` — o maior ganho estrutural ao desaninhar a pirâmide na camada de controller. ([expressjs.com/migrating-5](https://expressjs.com/en/guide/migrating-5.html))
- Para SQLite, prefira um driver com prepared statements: `better-sqlite3` é a escolha madura de produção; `node:sqlite` é o built-in sem dependências mas ainda Release Candidate (stability 1.2, **não** totalmente estável, RC só no v25.7.0, API síncrona via `DatabaseSync`/`StatementSync`); a antiga lib `sqlite3` por callbacks está superada e é a que se aposenta. Trade-offs: `better-sqlite3` = mais rápido, síncrono, testado em batalha, mas dependência nativa; `node:sqlite` = zero dependência, mas síncrono-only (pode bloquear o event loop sob carga). ([nodejs.org/api/sqlite](https://nodejs.org/api/sqlite.html))
- O Express 5 (exige Node.js 18+) traz quebras de roteamento e API que um god-class 4.x vai esbarrar no split MVC: path-to-regexp v8 (wildcard nomeado `/*splat`, segmentos opcionais com chaves `/:file{.:ext}`, sem RegExp inline), `req.query` agora é getter read-only, `extended` default `false`, `express.static` dotfiles default `ignore`, `req.body` undefined sem body parser, `res.status()` só 100–999. Removidos/renomeados: `app.del()`, `req.param(name)`, `res.sendfile()`, `res.send(status_number)`, as ordens `res.json(obj,status)`/`res.send(body,status)`/`res.redirect(url,status)`, `express.static.mime`. Codemod oficial: `npx codemod@latest @expressjs/v5-migration-recipe`. ([expressjs.com/migrating-5](https://expressjs.com/en/guide/migrating-5.html))
- Substituir qualquer hashing de senha caseiro por Argon2id (primeira escolha atual da OWASP: 19 MiB, 2 iterações, paralelismo 1); bcrypt só para sistemas legados (work factor ≥ 10, limite de 72 bytes, mitigação de *shucking* com base64-HMAC). Crypto caseira é perigosa: hashing seguro exige sal, dureza de memória/CPU, comparação em tempo constante e resistência a GPU/ASIC/side-channel/shucking — hashes rápidos e esquemas próprios não dão nada disso, e a OWASP manda usar funcionalidade vetada, não rolar a sua. ([cheatsheetseries.owasp.org/Password_Storage](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html))
- Usar prepared/parameterized statements em toda query; nunca concatenar input em SQL. Parameterized queries são a defesa primária inegociável: o engine compila a estrutura e liga valores como dados. `node:sqlite` (`db.prepare('... WHERE id = ?').get(id)` ou `:id`) e `better-sqlite3` fornecem isso nativamente, e o guia de segurança do Express recomenda explicitamente parameterized queries. ([nodejs.org/api/sqlite](https://nodejs.org/api/sqlite.html))
- Nunca logar segredos, credenciais, identificadores de sessão ou dados de cartão; mascarar, hashear ou omitir. A lista "data to exclude" da OWASP Logging Cheat Sheet proíbe em logs: senhas de autenticação, valores de sessão (hashear se precisar correlacionar), tokens de acesso, dados de conta/cartão, strings de conexão, chaves de criptografia e PII sensível. Regra: "never log data unless it is legally sanctioned"; campos sensíveis devem ser removidos, mascarados, hasheados ou criptografados antes de logar. ([cheatsheetseries.owasp.org/Logging](https://cheatsheetseries.owasp.org/cheatsheets/Logging_Cheat_Sheet.html))
- Endurecer a API com helmet mais a camada padrão (TLS, rate limiting, desabilitar `X-Powered-By`, `npm audit`, cookies seguros). Um único `app.use(helmet())` define 13 headers por padrão (CSP, COOP same-origin, CORP same-origin, Referrer-Policy no-referrer, HSTS, X-Content-Type-Options nosniff, X-Frame-Options SAMEORIGIN, remoção de X-Powered-By, X-XSS-Protection 0, etc.; COEP **não** vem ligado). Registre helmet antes das rotas; adicione TLS, rate limiting (mais estrito em auth), `app.disable('x-powered-by')`, cookies secure+httpOnly com nome de sessão não-padrão, validação de input e `npm audit`/Snyk. ([github.com/helmetjs/helmet](https://github.com/helmetjs/helmet))

## Autoria de Agent Skills do Claude Code (Anthropic)

Restrições e padrões oficiais para escrever a SKILL.md e seus arquivos de referência: frontmatter, descoberta por `description`, disclosure progressivo, concisão, graus de liberdade, desenvolvimento orientado a avaliação e portabilidade.

- Uma Skill é um diretório com a SKILL.md como entrypoint obrigatório; só o frontmatter YAML é obrigatório e `description` é o campo mais importante para descoberta ("Recommended so Claude knows when to use the skill"). Sem `name`, ele assume o nome do diretório; o comando `/skill-name` vem do diretório, não do `name`. Layout: SKILL.md (obrigatório) mais `template.md`, `examples/`, `scripts/` e docs de referência opcionais. ([code.claude.com/docs/skills](https://code.claude.com/docs/en/skills))
- O campo `name` tem limites rígidos: máx. 64 caracteres, só minúsculas/números/hífens, sem tags XML, e as palavras reservadas 'anthropic' e 'claude' são proibidas. Forma recomendada: gerúndio (`processing-pdfs`, `analyzing-spreadsheets`); evite nomes vagos (`helper`, `utils`, `tools`, `data`). ([platform.claude.com/.../best-practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices))
- O `description` deve ser não-vazio, máx. 1024 caracteres, sem XML, em terceira pessoa, e dizer o QUE a skill faz E QUANDO usá-la (com termos-gatilho). Use terceira pessoa ("Processes Excel files and generates reports"), nunca primeira/segunda, porque é injetado no system prompt. Inclua frases-gatilho concretas, tipos de arquivo e exemplos de pedido. ([platform.claude.com/.../best-practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices))
- No Claude Code, o texto combinado `description` + `when_to_use` é truncado em 1.536 caracteres na listagem de skills para economizar contexto — coloque o caso de uso principal primeiro. Esse cap de exibição (configurável via `maxSkillDescriptionChars`; orçamento ~1% do contexto via `skillListingBudgetFraction`) é camada separada do limite de validação de 1024 do campo. ([code.claude.com/docs/skills](https://code.claude.com/docs/en/skills))
- SKILL.md é um **prompt persistente** (entra no contexto e fica pelos turnos seguintes); arquivos de referência são **conhecimento de domínio sob demanda** (lidos do filesystem só quando precisos, custo zero de contexto até serem lidos). Essa divisão é a decisão central de autoria: escreva instruções permanentes e concisas no SKILL.md; jogue specs, datasets e exemplos grandes em arquivos separados. Scripts são executados (não carregados): só a saída consome tokens. ([code.claude.com/docs/skills](https://code.claude.com/docs/en/skills))
- Disclosure progressivo é o princípio arquitetural central, em três níveis: (1) metadados name+description sempre pré-carregados (~50–100 tokens/skill), (2) corpo da SKILL.md carregado quando relevante, (3) arquivos vinculados carregados só sob demanda. Anthropic compara a "a well-organized manual that starts with a table of contents, then specific chapters, and finally a detailed appendix"; com filesystem e execução, o contexto agregado é "effectively unbounded". ([anthropic.com/engineering/agent-skills](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills))
- Concisão é chave: mantenha o corpo da SKILL.md abaixo de 500 linhas e divida em referências ao se aproximar do limite. "The context window is a public good" — assuma que Claude já é muito inteligente e só adicione contexto que ele ainda não tem; questione o custo em tokens de cada parágrafo. ([platform.claude.com/.../best-practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices))
- Ao dividir em referências: mantenha-as exatamente um nível abaixo da SKILL.md, organize por domínio e adicione um índice a qualquer referência com mais de 100 linhas. Evite aninhamento profundo (Claude pode ler parcialmente, ex. `head -100`); use nomes descritivos (`form_validation_rules.md`, não `doc2.md`) e sempre barras normais, nunca backslash do Windows. ([platform.claude.com/.../best-practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices))
- Combine graus de liberdade com a fragilidade da tarefa: alta liberdade (instruções em texto) para trabalho aberto; baixa liberdade (scripts/comandos exatos) para operações frágeis e críticas em consistência ("Run exactly this script... Do not modify the command"). Liberdade média usa pseudocódigo/scripts parametrizados. Ofereça um default único com escape hatch em vez de muitas opções, e use terminologia consistente. ([platform.claude.com/.../best-practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices))
- Construa avaliações ANTES de escrever documentação extensa, com desenvolvimento orientado a avaliação e um loop Claude-A-autora / Claude-B-testa: identifique lacunas rodando tarefas representativas sem a skill, crie ≥ 3 cenários de teste, estabeleça baseline, escreva o mínimo de instruções para passar, itere. No Claude Code, o plugin `skill-creator` automatiza a comparação com/sem skill e o tuning de descrição. Teste em todos os modelos previstos (Haiku/Sonnet/Opus). ([platform.claude.com/.../best-practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices))
- Reusabilidade e desacoplamento vêm do padrão aberto Agent Skills, da composabilidade e da portabilidade auto-contida: skills seguem o Agent Skills open standard (agentskills.io), funcionam em Claude.ai/Claude Code/Agent SDK/Developer Platform e podem ser carregadas simultaneamente — então uma skill deve ser composável. Portabilidade exige declarar dependências explicitamente, usar nomes de tool MCP qualificados (`ServerName:tool_name`), referenciar scripts via `${CLAUDE_SKILL_DIR}`, usar barras normais e evitar conteúdo datado (use uma seção "old patterns"). ([anthropic.com/engineering/agent-skills](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills))
- Eficiência de tokens sob compactação: o Claude Code reanexa cada skill invocada após o resumo de contexto, mantendo os primeiros 5.000 tokens de cada uma, dentro de um orçamento combinado de 25.000 tokens. A auto-compactação preenche começando pela skill mais recente, então skills antigas podem cair — recompensa manter as instruções de maior valor nos primeiros ~5.000 tokens da SKILL.md. ([code.claude.com/docs/skills](https://code.claude.com/docs/en/skills))
- O Claude Code adiciona frontmatter de controle de invocação/execução sobre o padrão: `disable-model-invocation`, `user-invocable`, `allowed-tools`, `disallowed-tools`, `context: fork`, `agent`, `paths`, `model`, `effort`. Use `disable-model-invocation: true` para workflows com efeito colateral disparados manualmente; `user-invocable: false` para conhecimento de fundo; `context: fork` para rodar como subagente isolado; `paths` (globs) para auto-ativar só em arquivos correspondentes. ([code.claude.com/docs/skills](https://code.claude.com/docs/en/skills))
- Escopos de distribuição definem fronteiras de reúso: pessoal (`~/.claude/skills`), projeto (`.claude/skills`, versionado), plugin (namespaced `plugin-name:skill-name`) e enterprise/managed. Ordem de override em colisão: enterprise > pessoal > projeto, e qualquer um sobrepõe uma skill bundled de mesmo nome. Segurança: instale skills só de fontes confiáveis e audite scripts/dependências. ([code.claude.com/docs/skills](https://code.claude.com/docs/en/skills))

Os arquivos `.claude/commands/` continuam funcionando e geram o mesmo `/name`, mas skills são a forma recomendada (diretório de referências, frontmatter de controle, invocação automática); em colisão de nome, a skill tem precedência. | [code.claude.com/docs/skills](https://code.claude.com/docs/en/skills) |

## Camadas canônicas de MVC para APIs web (Flask e Express)

Como Model / Route-View / Controller / Service / DI / composition-root / error-handling / config mapeiam para Flask e Express, resolvendo armadilhas de nomenclatura e o debate fat-model vs fat-controller.

- MVC canônico divide em três responsabilidades: Model (dados + regras), View (apresentação), Controller (responde à entrada e media entre os dois). Como MVC nasceu como padrão de UI, num backend API os três papéis devem ser **relidos**, não copiados literalmente. ([en.wikipedia.org/MVC](https://en.wikipedia.org/wiki/Model%E2%80%93view%E2%80%93controller))
- Numa API backend, a "View" é serialização (ex.: `res.json` / marshalling JSON), **não** um template HTML; muitas APIs Express não têm objeto View algum. A responsabilidade de View colapsa em serialização de saída na borda do controller — não introduza camada de template numa API JSON. ([developer.mozilla.org/Express/routes](https://developer.mozilla.org/en-US/docs/Learn_web_development/Extensions/Server-side/Express_Nodejs/routes))
- Uma Route liga verbo HTTP + padrão de URL a um handler e deve ter **zero** regra de negócio; no Express, o Router declara rotas apontando para funções de controller ("A route is a section of Express code that associates an HTTP verb..."). Wiring canônico: controllers em `/controllers/*.js`, `/routes/*.js` importando-os, `app.use('/catalog', catalogRouter)`. ([developer.mozilla.org/Express/routes](https://developer.mozilla.org/en-US/docs/Learn_web_development/Extensions/Server-side/Express_Nodejs/routes))
- **Armadilha de precisão:** a "view function" do Flask é o route handler (algo de controller), **não** a "View" do MVC. Em termos MVC, ela faz o papel Route+entrada-de-Controller; a serialização JSON (`jsonify` / schema marshmallow) é a "View" real. Blueprints são a unidade modular que registra um subconjunto de rotas, análoga ao Router do Express, anexada no application factory. ([flask.palletsprojects.com/blueprints](https://flask.palletsprojects.com/en/stable/blueprints/))
- Controllers devem ser **finos**: parse/valida o request, chama um service, formata a resposta. Devem ter o mínimo de lógica por serem a camada mais difícil de testar ("Controllers are functions that separate out the code to route requests from the code that actually processes requests"); regra de negócio no controller acopla a HTTP/apresentação. ([developer.mozilla.org/Express/routes](https://developer.mozilla.org/en-US/docs/Learn_web_development/Extensions/Server-side/Express_Nodejs/routes))
- Models são a camada de acesso a dados/persistência (ORM: SQLAlchemy, Sequelize, Mongoose) e podem carregar comportamento de domínio; o termo "Model" conflaciona persistência e domínio. A boa prática separa: repositórios/ORM para I/O, objetos de domínio ricos para comportamento. ([oneuptime.com/express-project-structure](https://oneuptime.com/blog/post/2026-02-02-express-project-structure/view))
- A Service Layer é a fronteira da lógica de aplicação: define as operações disponíveis e coordena a resposta da app a cada uma, entre controllers e domínio. Fowler: "defines an application's boundary with a layer of services that establishes a set of available operations and coordinates the application's response in each operation." Ela detém controle de transação, coordenação cross-resource e orquestração. ([martinfowler.com/serviceLayer](https://martinfowler.com/eaaCatalog/serviceLayer.html))
- Regras de negócio (validações, cálculos, invariantes) pertencem aos objetos de domínio, não empilhadas em services; design "tudo em services" é o anti-padrão Anemic Domain Model. Fowler: objetos "little more than bags of getters and setters" "incur all of the costs of a domain model, without yielding any of the benefits" — "If all your logic is in services, you've robbed yourself blind." O service deve ser **fino** e coordenar; o domínio é "the heart of business software". ([martinfowler.com/AnemicDomainModel](https://martinfowler.com/bliki/AnemicDomainModel.html))
- Resolução fat-controller vs fat-model: prefira fat model a fat controller, mas a forma escalável é **controller fino + service fino (lógica de aplicação) + domínio rico**. Route = zero lógica; controller = fino (request→service→serialize); service = orquestração/transação/coordenação; domínio/model = regras/cálculos/validações. ([learn.microsoft.com/web-application-architectures](https://learn.microsoft.com/en-us/dotnet/architecture/modern-web-apps-azure/common-web-application-architectures))
- Dependency Injection: o código de aplicação usa constructor injection e nunca se auto-compõe; o grafo de objetos é montado só no Composition Root, perto do entry point. Seemann: "a (preferably) unique location in an application where modules are composed together" — "A DI Container should only be referenced from the Composition Root." Só aplicações têm Composition Root; bibliotecas/frameworks não. ([blog.ploeh.dk/CompositionRoot](https://blog.ploeh.dk/2011/07/28/CompositionRoot/))
- Mapeado aos frameworks: o `create_app()` do Flask é o composition root; o `app.js`/`server.js` do Express é o composition root. No Flask o factory constrói a instância, carrega config, chama `extension.init_app(app)` e `app.register_blueprint(bp)`; no Express o entry file importa routers, constrói/injeta services/repositories, monta routers com `app.use()` e sobe o listener. Mantenha o wiring fora dos módulos folha. ([flask.palletsprojects.com/appfactories](https://flask.palletsprojects.com/en/stable/patterns/appfactories/))
- Tratamento de erro centralizado no Express é um middleware de 4 args `(err, req, res, next)` registrado **por último**; handlers encaminham falhas com `next(err)` ("You define error-handling middleware last"; "If you pass anything to the next() function (except the string 'route'), Express regards the current request as being an error"). Use uma classe de erro custom para padronizar status+corpo; se `res.headersSent`, delegue com `return next(err)`. ([expressjs.com/error-handling](https://expressjs.com/en/guide/error-handling.html))
- O Flask centraliza erros via `@app.errorhandler` / `app.register_error_handler` e subclasses de `HTTPException`, registrados no factory: levanta `abort(404)` ou exceções custom nas view functions/services e registra handlers (ex.: `@app.errorhandler(404)`) em `create_app()`, espelhando o handler único do Express. ([flask.palletsprojects.com/errorhandling](https://flask.palletsprojects.com/en/stable/errorhandling/))
- Config pertence a um módulo dedicado carregado no composition root, com valores por ambiente; nunca hardcode segredos. Flask: classe Config base com subclasses Development/Testing/Production via `app.config.from_object()` (e `from_pyfile`/instance folder para segredos reais). Express: módulo `config/` lendo `process.env` via dotenv. Config por ambiente é o que viabiliza o benefício de instâncias isoladas de teste do factory. ([flask.palletsprojects.com/appfactories](https://flask.palletsprojects.com/en/stable/patterns/appfactories/))
- Direção de dependência: Routes → Controllers → Services → Domain/Models → persistência; dependências apontam para dentro e nunca pulam de volta para fora. Um controller nunca consulta o DB direto e um model nunca importa o objeto request HTTP. Esse fluxo unidirecional é o que torna o service reusável fora do HTTP e cada camada testável isoladamente. ([oneuptime.com/express-project-structure](https://oneuptime.com/blog/post/2026-02-02-express-project-structure/view))

| Antigo | Moderno | Por quê | Fonte |
| --- | --- | --- | --- |
| Express 4: encaminhar erros async à mão com try/catch + `next(err)` em cada handler, ou `express-async-errors` / wrapper `catchAsync` | Express 5: handlers/middleware que retornam Promise chamam `next(value)` automaticamente ao rejeitar/lançar, levando o erro ao handler centralizado | Express 5 embutiu a propagação de erro async no framework, eliminando o try/catch por handler e os wrappers de terceiros que o Express 4 exigia. | [expressjs.com/error-handling](https://expressjs.com/en/guide/error-handling.html) |

## OWASP API Security Top 10 (2023) + OWASP Top 10:2021 — mapeamento de vulnerabilidades

Mapeamento de defeitos típicos de app legado para as categorias OWASP (web 2021 e API 2023), com a orientação atual de hashing de senhas. Útil para classificar achados no catálogo de anti-padrões.

- SQL injection por concatenação de string → Web: **A03:2021 Injection** (CWE-89). Atenção: Injection **não** é categoria isolada no OWASP API Security Top 10 2023 (era API8:2019, removida em 2023) — para APIs cite a web A03:2021. As categorias API 2023 vão de API1 BOLA a API10 Unsafe Consumption of APIs. Remediação: parameterized queries/prepared statements em todo SQL dinâmico, ORM seguro, validação allowlist server-side, conta de DB com menor privilégio; escaping de saída só como fallback. ([owasp.org/A03_2021-Injection](https://owasp.org/Top10/2021/A03_2021-Injection/))
- Credenciais/chaves hardcoded → Web: **A07:2021 Identification and Authentication Failures** (CWE-798) para credenciais de aplicação; **A02:2021 Cryptographic Failures** (CWE-321) para chaves de cripto/assinatura hardcoded. API: **API8:2023 Security Misconfiguration**. Remediação: remova segredos de código/config versionados, carregue de env ou secret manager (Vault, AWS/GCP/Azure), config por ambiente, rotacione/revogue tudo que já foi commitado (assuma comprometido), adicione scanning (gitleaks, trufflehog) e pre-commit hooks. ([owasp.org/A07_2021](https://owasp.org/Top10/2021/A07_2021-Identification_and_Authentication_Failures/))
- Senha em texto puro e hashing caseiro/fraco → Web: **A02:2021 Cryptographic Failures** (CWE-916, CWE-327). API: **API2:2023 Broken Authentication** e/ou **API8:2023 Security Misconfiguration**. O texto de prevenção do A02:2021 diz: "Store passwords using strong adaptive and salted hashing functions with a work factor (delay factor), such as Argon2, scrypt, bcrypt or PBKDF2." Remediação: nunca texto puro/cripto reversível; substitua hashes rápidos (MD5/SHA-1/SHA-256) e esquemas próprios por hash adaptativo memory-hard (Argon2id preferido) com sal único + pepper opcional; faça rehash transparente no próximo login bem-sucedido. ([owasp.org/A02_2021](https://owasp.org/Top10/2021/A02_2021-Cryptographic_Failures/))
- Endpoint admin de SQL arbitrário (executa SQL do cliente) → Web: **A03:2021 Injection** E **A04:2021 Insecure Design**. API: **API5:2023 Broken Function Level Authorization** (e A01:2021 Broken Access Control). É achado duplo: executar SQL controlado pelo atacante é A03; a própria existência do endpoint é A04 (controle ausente por design). Remediação: **delete o endpoint** — não há jeito seguro de expor SQL arbitrário; se admins precisam de operações, ofereça um conjunto pequeno de ações predefinidas e parametrizadas atrás de authn forte + RBAC, conta de DB com menor privilégio e audit logging. ([owasp.org/A04_2021-Insecure_Design](https://owasp.org/Top10/2021/A04_2021-Insecure_Design/))
- Segredo vazado em `/health` (ou actuator/debug) → Web: **A05:2021 Security Misconfiguration**. API: **API8:2023 Security Misconfiguration** (também toca A01:2021 por exposição não-autenticada e A02:2021 pela exposição do dado). Remediação: health endpoints devem retornar só status mínimo (UP/DOWN), nunca config/segredos; restrinja diagnósticos detalhados a chamadores internos autenticados ou desabilite em produção; audite defaults de framework. ([owasp.org/A05_2021](https://owasp.org/Top10/2021/A05_2021-Security_Misconfiguration/))
- Senhas (ou hashes) retornados numa resposta de listagem de usuários → Web: **A02:2021 Cryptographic Failures** (exposição de dado sensível). API: **API3:2023 Broken Object Property Level Authorization** (que subsume a antiga "Excessive Data Exposure"). Remediação: nunca inclua campos de senha/hash em resposta; não confie no cliente para filtrar; use DTOs de resposta / allowlists de serialização; anote campos sensíveis para exclusão (ex.: `@JsonIgnore` / write-only); aplique autorização nível-propriedade. ([owasp.org/API-Security/2023](https://owasp.org/API-Security/editions/2023/en/0x11-t10/))
- Números de cartão (PAN) gravados em logs → Web: **A09:2021 Security Logging and Monitoring Failures** (CWE-532), com **A02:2021** como consequência de exposição. API: **API8:2023 Security Misconfiguration**. Logar PAN é falha de logging (A09) e exposição (A02) e viola PCI DSS (proíbe armazenar PAN completo e CVV). Remediação: nunca logar PAN/CVV/track data; mascarar para first6/last4 (ou só last4) na camada de log; filtros de redação field-level; tokenizar; limpar/rotacionar logs existentes; restringir e monitorar acesso a logs. ([owasp.org/A09_2021](https://owasp.org/Top10/2021/A09_2021-Security_Logging_and_Monitoring_Failures/))
- Falta de authn/authz em endpoints admin destrutivos (delete-all, reset, bulk-update) → Web: **A01:2021 Broken Access Control** (+ A07:2021 se não há authn alguma). API: **API5:2023 Broken Function Level Authorization** (+ API2:2023 se não-autenticado). Remediação: imponha authn E checagem de papel/permissão server-side em todo endpoint admin e de mudança de estado; deny-by-default; centralize o controle de acesso; nunca confie em URL oculta ou botão client-side; exija confirmação e audit log para operações destrutivas; menor privilégio. ([owasp.org/A01_2021](https://owasp.org/Top10/2021/A01_2021-Broken_Access_Control/))
- Dados órfãos no delete (sem cascade / sem transação) → Web: **A04:2021 Insecure Design** (encaixe conceitual mais próximo, **não** um match forte de CWE). Sem categoria direta no API Top 10 2023. **Mapeamento de baixa confiança, marcado deliberadamente.** É primariamente defeito de integridade/confiabilidade. Aviso importante: apesar do nome, **A08:2021 Software and Data Integrity Failures NÃO se aplica** (A08 é integridade de CI/CD, desserialização insegura, auto-updates não-assinados, não integridade referencial de DB). Remediação: envolva deletes multi-passo numa transação única (atomicidade/rollback); declare FKs com `ON DELETE CASCADE` ou limpeza determinística dos filhos; considere soft-delete; imponha integridade referencial no DB; testes de regressão garantindo zero linhas órfãs. ([owasp.org/A04_2021-Insecure_Design](https://owasp.org/Top10/2021/A04_2021-Insecure_Design/))
- Hashing seguro recomendado atual: **Argon2id** com config mínima m=19456 KiB (19 MiB), t=2, p=1 — preferido sobre scrypt, bcrypt e PBKDF2 (nessa ordem). Ordem de preferência: (1) Argon2id [tunings alternativos: m=47104/t=1, m=12288/t=3, m=9216/t=4, m=7168/t=5, todos p=1]; (2) scrypt N=2^17, r=8, p=1 quando Argon2id indisponível; (3) bcrypt work factor ≥ 10, limite de 72 bytes (pré-hash com `bcrypt(base64(hmac-sha384(password, pepper)))`); (4) PBKDF2 só para FIPS-140 (HMAC-SHA256: 600.000; HMAC-SHA512: 220.000; HMAC-SHA1 legado: 1.400.000). Sempre sal único por senha; pepper separado é defesa em profundidade opcional. ([cheatsheetseries.owasp.org/Password_Storage](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html))

| Antigo | Moderno | Por quê | Fonte |
| --- | --- | --- | --- |
| Senha em texto puro (ou criptografia reversível de senha) | Hashing Argon2id mínimo m=19456 KiB (19 MiB), t=2, p=1 com sal único por senha | Texto puro/reversível expõe toda credencial em qualquer leitura/backup; A02:2021 exige hashing adaptativo, salgado e lento. | [cheatsheetseries.owasp.org/Password_Storage](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html) |

## Recomendações consolidadas para o catálogo e o playbook

### (a) Seção de APIs depreciadas do catálogo de anti-padrões

Catalogar cada item como uma regra detectável "antigo → moderno → por quê → fonte", agrupada por stack:

- **Python/Flask/Werkzeug:** `datetime.utcnow()`/`utcfromtimestamp()` → `datetime.now(timezone.utc)`/`fromtimestamp(ts, tz=timezone.utc)` ([datetime](https://docs.python.org/3/library/datetime.html)); `werkzeug.urls.url_decode/url_encode` → `urllib.parse` ([werkzeug](https://werkzeug.palletsprojects.com/en/stable/changes/)); `flask.Markup/escape` → `markupsafe`; `@app.before_first_request` → setup no `create_app()`; `flask.json.JSONEncoder` → `DefaultJSONProvider` em `app.json`; `flask.__version__` → `importlib.metadata.version('flask')` ([flask changes](https://flask.palletsprojects.com/en/stable/changes/)).
- **Node/Express:** `app.del`→`app.delete`, `res.sendfile`→`res.sendFile`, `res.send(body,status)`→`res.status(status).send(body)`, `res.send(200)`→`res.sendStatus(200)`, `req.param('id')`→`req.params/body/query`, `express.static.mime`→`mime-types`, wildcard `'/*'`→`'/*splat'` ([Express 5 migration](https://expressjs.com/en/guide/migrating-5.html)); lib `sqlite3` callbacks → `better-sqlite3`/`node:sqlite` ([node:sqlite](https://nodejs.org/api/sqlite.html)).
- **Transversal (vira regra de severidade alta):** SQL por string-format → placeholders parametrizados ([sqlite3](https://docs.python.org/3/library/sqlite3.html)); hash de senha caseiro/rápido (MD5/SHA-*) → Argon2id; `SECRET_KEY`/segredos hardcoded → variáveis de ambiente ([Password Storage](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html), [A02:2021](https://owasp.org/Top10/2021/A02_2021-Cryptographic_Failures/)).
- **Meta-skill:** sinalizar arquivos `.claude/commands/<name>.md` como forma legada → migrar para `.claude/skills/<name>/SKILL.md` ([skills](https://code.claude.com/docs/en/skills)).
- Para cada anti-padrão de segurança, anexar a tag OWASP correspondente (A03/A07/A02/A04/A05/A09/A01:2021 e API2/API3/API5/API8:2023) para que o catálogo classifique severidade de forma consistente; marcar explicitamente mapeamentos de baixa confiança (ex.: dados órfãos → A04:2021, **não** A08).

### (b) Transformações do refactor playbook

- **Substituições de API:** rodar a auditoria de imports/chamadas depreciadas acima como passo 1; no Express, aplicar o codemod oficial `npx codemod@latest @expressjs/v5-migration-recipe` antes de quebrar o god-class. Rodar testes com `python -W error::DeprecationWarning` para falhar em uso legado remanescente.
- **Datas:** padronizar UTC timezone-aware em models/serializers/colunas (`DateTime(timezone=True)` onde o dialeto suportar), tratando o round-trip naive como nota de implementação, não como motivo para manter `utcnow()`.
- **SQL:** centralizar acesso a dados num repositório/Model e converter todo SQL concatenado em prepared statements parametrizados (`?`/`:name`); controllers nunca tocam SQL cru. Deletar por completo qualquer endpoint de SQL arbitrário ([A04:2021](https://owasp.org/Top10/2021/A04_2021-Insecure_Design/)).
- **Async:** converter a pirâmide de callbacks para async/await; no Express 5 remover `.catch(next)` (rejeições auto-encaminham); `util.promisify` para callbacks error-first remanescentes.
- **Senhas:** trocar storage por Argon2id (m=19 MiB, t=2, p=1) via lib vetada (`argon2-cffi`/`node-argon2`); manter bcrypt só se um store legado obrigar (cost ≥ 10, mitigação 72-byte/base64-HMAC), com rehash transparente no próximo login. Nunca comparar com `==`; usar verificação em tempo constante.
- **Segredos & debug:** mover `SECRET_KEY`/credenciais para env (`secrets.token_hex()`), `from_object`/`from_prefixed_env` (Flask) ou `config/` + dotenv (Express); debug de `FLASK_DEBUG` default `False` com fail-fast em produção; nunca `app.run(debug=True)`. Adicionar scanning (gitleaks/trufflehog) e rotacionar segredos já commitados.
- **Logs:** inserir passo de redação (mascarar senhas, tokens, session IDs, connection strings, PAN/PII) — logger estruturado com redaction paths (ex.: pino); remover `console.log`/debug que ecoa request bodies ou credenciais ([Logging](https://cheatsheetseries.owasp.org/cheatsheets/Logging_Cheat_Sheet.html), [A09:2021](https://owasp.org/Top10/2021/A09_2021-Security_Logging_and_Monitoring_Failures/)).
- **Camada de segurança (Express):** `app.use(helmet())` e `app.disable('x-powered-by')` antes das rotas, rate limiter global + mais estrito em auth, limites de body, validação de input (zod/express-validator).
- **Respostas:** introduzir DTOs/allowlists de serialização para nunca devolver campos de senha/hash; impor autorização nível-propriedade ([API3:2023](https://owasp.org/API-Security/editions/2023/en/0x11-t10/)).
- **Deletes:** envolver deletes multi-entidade numa transação única + FK cascade/limpeza determinística, com teste de regressão garantindo zero órfãos.
- **Processo:** dirigir toda a refatoração pelo loop TDD red-green-refactor obrigatório (um teste falhando por comportamento — rota, hash, query — antes de extrair do god-class) e manter README/ROADMAP em sincronia no mesmo commit; adicionar `npm audit` no CI.

### (c) Restrições de autoria da skill

- **Frontmatter:** `name` em kebab-case minúsculo (≤ 64 chars), preferir gerúndio (ex.: `refactoring-projects`), sem `claude`/`anthropic` nem XML; `description` em terceira pessoa, ≤ 1024 chars, dizendo O QUE e QUANDO com gatilhos concretos ("refactor", "modernize", nomes de framework, tipos de arquivo), caso de uso principal primeiro para sobreviver ao truncamento de 1.536 chars do Claude Code.
- **Estrutura:** corpo da SKILL.md como instruções permanentes concisas (alvo < 500 linhas, maior valor nos primeiros ~5.000 tokens por causa da reanexação sob compactação); empurrar specs/exemplos/padrões por linguagem para arquivos de referência **um nível abaixo**, organizados por domínio (`reference/python.md`, `reference/express.md`, `reference/owasp.md`), com índice em qualquer referência > 100 linhas e nomes descritivos.
- **Graus de liberdade:** scripts/comandos exatos para passos frágeis/destrutivos (ex.: migração de hashes, deletes transacionais); texto de alto nível para julgamento aberto (qual camada recebe a lógica). Um default com escape hatch, terminologia consistente.
- **Portabilidade/desacoplamento:** declarar dependências explicitamente, nomes de tool MCP qualificados (`ServerName:tool_name`), referenciar scripts via `${CLAUDE_SKILL_DIR}`, barras normais, evitar conteúdo datado (usar seção "old patterns" para as APIs depreciadas em vez de "em 2026..."), assumir coexistência com outras skills.
- **Avaliação:** desenvolvimento orientado a avaliação — ≥ 3 cenários (ex.: detectar `utcnow()`, SQL concatenado, senha em texto puro), baseline sem skill, mínimo de instruções para passar, loop Claude-A-autora/Claude-B-testa (ou `skill-creator`), testar em Haiku/Sonnet/Opus.
- **Distribuição:** skill de projeto versionada em `.claude/skills/<name>/SKILL.md`; usar `paths:` globs para auto-ativar só em arquivos relevantes; `disable-model-invocation`/`user-invocable` deliberados para qualquer ação com efeito colateral.

### (d) Arquitetura-alvo MVC

- **Fluxo unidirecional:** Routes → Controllers → Services → Domain/Models → persistência; dependências só apontam para dentro. Controller nunca consulta DB direto; domínio/model nunca importa o request HTTP — é isso que mantém service+domínio agnósticos de framework e testáveis sem HTTP.
- **Routes/View:** declarações de rota sem lógica (Express `/routes` + mount no `app.js`; Flask Blueprints registrados no `create_app()`). A "View" da API é serialização JSON (`res.json`/`jsonify`/schema marshmallow), **não** template HTML. Lembrar a armadilha: a "view function" do Flask é o handler de rota (papel de controller), não a View do MVC.
- **Controllers finos:** parse/valida o request, chama exatamente uma operação de service, serializa a resposta e mapeia erros para status — zero regra de negócio.
- **Services (lógica de aplicação):** orquestração, transações e coordenação cross-entidade; uma operação por caso de uso, reusável por controller, CLI ou worker. Manter o service fino — ele coordena.
- **Domain/Models (lógica de domínio):** validações, cálculos e invariantes nos objetos de domínio, evitando o Anemic Domain Model; separar persistência (ORM/repositório: SQLAlchemy, Sequelize, Mongoose) do comportamento de domínio mesmo quando ambos são chamados de "model". Alvo escalável explícito: **controller fino + service fino + domínio rico** (preferir fat model a fat controller, mas não despejar tudo na classe ORM).
- **Composition root:** montar o grafo de objetos uma vez, na borda. Flask `create_app()` carrega config, roda `extension.init_app(app)` e registra blueprints/error handlers; Express `app.js`/`server.js` constrói e injeta services/repositories, monta routers e sobe o listener. Módulos folha usam constructor injection e nunca referenciam um DI container.
- **Erro centralizado:** um handler no composition root. Express: middleware de 4 args `(err,req,res,next)` **depois** de todas as rotas, com classe de erro custom para status+shape (Express 5 auto-encaminha rejeições async). Flask: `@app.errorhandler` / subclasses de `HTTPException` no `create_app()`. Controllers lançam/raise; nunca montam payload de erro.
- **Config:** módulo dedicado por ambiente (Flask Config Dev/Test/Prod via `from_object`; Express `config/` + dotenv), carregado no entry point e injetado; segredos em env/instance folder, nunca no código — config por ambiente é o que habilita o benefício de instâncias isoladas de teste do factory.
- **Layout Flask sugerido:** `app/__init__.py` (`create_app`), `app/models/` (SQLAlchemy), controllers como view functions de Blueprint por feature, `app/services/`, `config.py`, e um `run.py`/`wsgi.py` fino.
